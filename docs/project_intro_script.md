# 專案介紹簡報講稿


## 標題頁

大家好,今天跟大家介紹我目前的研究專案。一句話講:當 LLM 要讀非常長的文件時,GPU 記憶體會被兩個東西吃掉——模型本身,和推論過程的 KV cache。我們的問題是:記憶體有限時,壓縮哪一邊比較划算?以及能不能設計出更聰明的 KV cache 壓縮方法。整個實驗在一張 24GB 的 RTX 4090 上進行,用 Qwen2.5-7B,目前進行到第四階段。


## 背景:長上下文推論的記憶體牆

先看問題有多真實。7B 模型的 bf16 權重約 14.3 GB,在 24GB 的卡上就吃掉六成。而 KV cache 是模型讀文件時的「筆記」,每讀一個 token 就要多存約 56KB,跟上下文長度成正比:64K token 的單一請求就要 3.5GB。換句話說,長上下文、多併發的場景下,KV cache 會反過來變成比權重更大的記憶體支出。所以「量化」不只該問怎麼壓權重,也該問怎麼壓 KV cache——這是兩個可以獨立轉的旋鈕。


## 兩個研究問題

專案有兩層問題。RQ1 是部署層:在真實 serving 引擎 vLLM 上,壓權重和壓 KV cache 各自換到多少容量、付出多少品質代價,兩者可不可以疊加。RQ2 是研究層:現有低 bit KV 量化在 2-bit 附近品質會崩,我們提出 Hybrid Polar-KIVI——Key 用極座標量化、Value 用 per-token 量化,目標是在約 3.3 bit 的預算下,品質接近 4.5 bit 的 int4,而不是像 2.5 bit 的 int2 一樣崩掉。


## 實驗設計:兩條 runtime 路線

方法上把實驗切成兩條路線。部署路線用 vLLM,量的是真實 serving 的速度和容量;研究路線用 HuggingFace transformers,以 simulated quantization 量品質影響——量化後立刻反量化,品質損失是真的,記憶體節省則用解析公式計算,絕不宣稱實測省了多少 VRAM。一條鐵則:兩條路線的速度數字永遠不放進同一張表,避免拿不同引擎的數字互比造成誤導。兩條路線共用同一批凍結 prompts 和同一套結果 schema。


## 可重現性紀律:每個數字都可追溯

這頁講我們怎麼防止自己「不小心作弊」。評測集只生成一次,SHA256 鎖進 manifest,程式裡直接 assert 禁止重生成;NIAH 的「針」是隨機十六進位密碼,模型不可能靠背訓練資料答對。所有報告數字必須追溯到 CSV 或 raw 檔;失敗、OOM 也是數據,照樣記錄。十筆以上的問題與 fallback 全部記在 ISSUES.md,包含對我們故事「不利」的結果。這套紀律的目的,是讓最後論文裡的每一格數字都經得起檢查。


## 品質怎麼量:三件套

品質評測用三個互補的工具。第一是 NIAH 大海撈針:在長文件的不同深度藏一個隨機密碼,問模型密碼是什麼——這直接測長上下文檢索能力,涵蓋中文、英文、程式碼三種語料,長度從 4K 到 63K、五種埋針深度。第二是 LongBench,八個子集涵蓋中英問答、檢索、程式碼補全,用官方 metrics 計分。第三是 cached PPL:讓長文先流過(量化的)KV cache 再算困惑度,確保 KV 量化的影響真的參與計分。三者角度不同,可以交叉驗證。


## Phase 路線圖與目前進度

整個計畫切成十個 phase。P0 建環境,P1 打 BF16 基準線,P2 做 weight-only 量化,P3 完成部署路線的 2×2 主表——這四個已完成並各有報告。目前在 P4,在 HF 路線建立 int4/int2 低 bit KV 的品質曲線,作為 P6 的對照組。P6 是研究主貢獻 Hybrid Polar-KIVI,P7 做場景分析,P9 彙整成最終報告。P5 和 P8 是可選項,不阻塞主線。


## 成果 1:Weight 量化幾乎是免費午餐(P1–P2)

第一個結論:壓權重非常划算。AWQ 4-bit 把權重從 14.3GB 壓到 5.2GB,省下的記憶體全部變成 KV cache 的空間,KV pool 容量變成 2.53 倍;而品質幾乎沒有代價——63K 的中英 NIAH 依然全對,LongBench 平均只從 0.493 掉到 0.479,長上下文吞吐甚至因為記憶體壓力變小而更快。GPTQ 結果相近。這條基準線同時證明我們的評測管線是健康的。


## 成果 2:FP8 KV 的反直覺發現(P3)

第二個發現比較戲劇性,也是目前最有價值的負面結果。理論上 FP8 KV 把容量再翻倍,AWQ 加 FP8 可達 5 倍 KV pool,帳面非常漂亮。但品質完全崩潰:63K 大海撈針從全對掉到接近全錯,LongBench 平均掉了六成。原因是 vLLM 0.10.2 的 FP8 KV 走的是舊的 V0/XFormers 路徑,加上動態 KV scale 無法保住長上下文檢索。教訓有兩個:第一,「容量提升」和「可用品質」必須分開驗證;第二,這正好證明了低 bit KV 量化需要更聰明的方法——這就是 P6 研究主線的動機。


## 進行中(P4):低 bit KV 品質曲線 — 現況與一個插曲

P4 的目標是在 HF 路線畫出 int4 和 int2 KV 的品質曲線,當 P6 的對照組。目前 fp16 對照組已完成,中英 NIAH 各 45 題全對,證明評測管線健康。插曲是:HF 內建 quanto int4 cache 跑出來全滅、輸出是亂碼——連 4K 都全錯。int4 KV 在 4K 理論上幾乎無損,所以這不是量化品質數據,而是管線 bug:root cause 指向 quanto cache 與 chunked prefill 的不相容。我們照紀律記錄進 ISSUES,並準備用自家已通過測試的 fake-quant 實作 int4_sim/int2_sim 作為更乾淨的對照組——殘差視窗、分組方式與 Hybrid 完全對齊,控制變因反而更乾淨。


## 研究主貢獻(P6):Hybrid Polar-KIVI

接著是研究主貢獻。Key cache 先做隨機正交旋轉,再把 channel 兩兩配對轉成極座標。妙處在數學:旋轉後每對座標近似獨立高斯,此時角度 θ 恰好是均勻分布——均勻量化器對它是理論最優;半徑 r 是 Rayleigh 分布,有封閉解可以放非均勻格點。Value 則用 KIVI 式 per-token 量化,最近 128 個 token 保持全精度。整體有效位元約 3.3 bit,卡在 int4 的 4.5 和 int2 的 2.5 之間。要驗證的假說是:品質接近 int4、遠好於 int2。另有 θ=3bit 與不旋轉的 ablation。所有 bit 帳都把 scale/zero-point 的 overhead 誠實計入。


## 下一步計畫

近期的路徑是:先修好 P4 的對照組——用自家 fake-quant 替代 quanto,補齊 int4/int2 的 NIAH 和 cached PPL;然後進入 P6 主實驗,跑 Hybrid 主配置和三組 ablation,回答「同 bit 預算下品質是否守住」;P7 做場景敏感度,看中文、程式碼、多文件檢索誰先受傷;最後 P9 彙整成報告。另外有幾個第一性原則延伸方向已經排進 backlog:Rayleigh 最優格點的 r 量化、pre-RoPE 座標系的極座標量化、以及 retrieval-head 感知的混合精度——最後這個可以直接解釋 P3 FP8 崩潰的機制,是最有論文潛力的延伸。


## 三個 Take-aways

最後三句話總結。第一,weight 量化接近免費午餐:AWQ 換到 2.5 倍容量,品質幾乎無損,部署上應該直接採用。第二,KV 量化的魔鬼在品質:帳面容量再漂亮,長上下文檢索崩掉就沒有意義,容量和品質必須一起驗收——這是我們用 FP8 實測換來的教訓。第三,我們的貢獻是把「更聰明的低 bit KV 量化」做成一個可檢驗的提案:極座標量化有分布上的理論依據,並配了一套凍結評測、bit 帳透明、失敗照記的實驗紀律。歡迎大家給意見,特別是評測設計和 ablation 的部分。謝謝!
